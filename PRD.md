# PRD — 수행 계획서 자동 작성 에이전트

## 1. 프로젝트 개요
- **문제**: 사람마다 품질·형식이 제각각이라 외부 제출 시 회사 표준 형식에 맞지 않는다.
- **배경**: 외부 제출 문서라 형식·완성도가 곧 회사 신뢰도로 직결되며, 제출 전 형식 검수·재작업이 매번 발생한다.
- **기대 변화**: 필수 값만 입력하면 표준 형식으로 몇 분 만에 완성 (현재 건당 작성 몇 시간 → 이후 몇 분).

## 2. 사용자 분석
- **주 사용자**: 공공·SI 사업 수주를 맡는 사업 담당 (동시 여러 건 진행).
- **사용 맥락**: 프로젝트 단계에서 외부 제출용 수행 계획서를 작성할 때.
- **현재 해결 방식과 한계**: 빈 양식 템플릿에 처음부터 채움 → 항목마다 무엇을 쓸지 몰라 막히고 시간이 오래 걸린다.

## 3. 기능 요구사항 (우선순위 순)
1. **필수 값 입력받기** — 아래 "필수 입력 값" A 그룹 8개를 받기
2. **내용 자동 작성** — 입력값 기반으로 막히던 항목까지 자동 채움
3. **표준 양식(.docx)으로 출력** — 회사 표준 워드 템플릿 서식을 보존한 채 값만 채워 생성
4. **수정·재생성** — 일부만 고쳐 다시 출력

- **다음 단계 기능**: 빠진 값·오류 체크, 출력 포맷 확장(HWP/PDF)
- **차별점**: 회사 표준 양식(.docx)·문체에 정확히 맞춰 작성돼, 검수 없이 그대로 제출 가능하다.

### 필수 입력 값 (기준 템플릿: `01.AgentGo PoC(가치 증명) 수행 계획서_v1.2.docx`)
이 템플릿은 빈 양식이 아니라 채워진 사례 문서이며, "양식 골격(고정)"과 "사업별 가변 값"을 분리해 다룬다.

**A. 필수 입력 (없으면 작성 불가) — 8개**
1. 고객사명 (`client_name`)
2. 착수일 (`start_date`)
3. 총 수행 주차 (`total_weeks`) — 2·3과 합쳐 일정 섹션 날짜 자동 산출
4. PoC 목적·배경 (`poc_purpose`) — 고객 업무 목표 한두 줄
5. 검증 과제 목록 (`verify_tasks[]`) — 예: RAG 정합성, 기간계 연계, M365 연계
6. 연동 대상 시스템 (`integration_targets`) — SSO 종류 / 기간계 유무 / M365 유무 등
7. 솔루션 구성 (`solution_config[]`) — 제품별 User 수·적용 LLM·vCPU
8. 고객 측 PM·승인권자 (`client_pm`) — 연락처는 마스킹 입력

**B. 선택 입력 (기본값 자동 적용, 협의 시 변경)**
- KPI 목표 수준(기본: 정확성 95%↑, 사용성 4.0↑), 인프라 OS(기본: Ubuntu 22.04 / 2안 Rocky), NAS 사용 여부, 회의 주기(기본: 격주), 단계 구성(기본: 6단계 표준 WBS)

**C. 고정값 (입력 불필요 — 템플릿에서 상속)**
- 인프라 표준 스펙, 포트·도메인 리스트, 클로잇 측 연락망 골격, 서명란

## 4. 비기능 요구사항
- **사용 환경**: 클로드 코드(CLI)에서 동작, 프로젝트 폴더 문서를 직접 처리.
- **출력 형식**: 회사 표준 워드 템플릿(.docx) 구조·서식을 보존한 채 값만 채워 .docx로 출력. (마크다운 초안만으로는 목적 미충족)
- **데이터 민감도**: 인력 개인정보 + 고객사 정보 포함 → 외부 LLM 원문 노출 주의 (입력 단계 마스킹 / 실명 대신 직무·등급으로 받기).
- **제약**: 예산(유료 API·도구 도입 제한) → .docx 처리는 오픈소스 라이브러리(예: python-docx) 전제. 표준 양식은 확보됨(위 기준 템플릿).

## 5. 성공 지표
- **핵심 지표**: 6개월 후 신규 수행 계획서의 70~80%가 이 에이전트로 작성 (생성 문서 메타 태그/워터마크로 자동 집계).
- **보조 지표**: 건당 작성 시간 단축, 제출 전 형식 반려·재작업 감소.

---

## 6. 사용자 여정 (User Journey)
1. 사업 담당이 클로드 코드에서 에이전트 호출
2. 에이전트가 필수 4값을 차례로 질문 → 담당이 입력 (개인정보·고객사명은 직무/코드로 입력하도록 유도)
3. 에이전트가 회사 표준 양식에 맞춰 전 항목 초안 자동 작성
4. 담당이 결과 확인 → "3번 일정만 다시" 식으로 부분 수정 요청
5. 재생성 → 최종본 확정, 메타 태그 포함해 파일로 저장/출력

## 7. 기능별 상세 사양 (입력 → 동작 → 출력)

### 필수 값 입력
- **입력**: 사업명·기간·범위/목적·주요 산출물
- **동작**: 누락 항목 되묻기, 민감정보는 마스킹 형태로 받기
- **출력**: 검증된 입력 값 세트

### 내용 자동 작성
- **입력**: 검증된 값 세트 + 기준 템플릿(.docx)
- **동작**: 가변 항목(개요·수행 범위·KPI·일정 WBS·R&R 등)을 표준 문체로 생성, 고정 항목은 템플릿에서 상속, 착수일 기준 일정 날짜 자동 산출
- **출력**: 표준 서식을 보존한 수행 계획서 .docx

### 수정·재생성
- **입력**: 대상 항목 + 수정 지시
- **동작**: 해당 항목만 재작성(나머지 보존)
- **출력**: 갱신된 계획서

## 8. 개발 우선순위
- **MVP**: 필수 값(A 8개) 입력 + 내용 자동 작성 + **표준 양식(.docx) 출력** (기준 템플릿: `01.AgentGo PoC(가치 증명) 수행 계획서_v1.2.docx` 1종)
- **다음**: 수정·재생성, 빠진 값·오류 체크, 선택 입력(B) 기본값/되묻기 처리
- **추후**: 표준 양식 다종 지원, 출력 포맷 확장(HWP/PDF), 메타 태그 자동 집계

## 9. 위험 요소
- 개인정보·고객사 정보의 외부 LLM 노출 → 입력 마스킹 규칙을 MVP 전에 확정.
- **템플릿 서식 보존**: .docx의 표·스타일·서식을 깨지 않고 값만 채워 넣는 것이 기술 난제 → 텍스트 치환이 아니라 구조(표 셀·플레이스홀더) 기반 주입 필요.
- 기준 템플릿이 빈 양식이 아닌 **채워진 사례**라 "고정 골격 / 가변 값" 구분이 모호하면 잘못된 값이 남을 수 있음 → 가변 항목 플레이스홀더화가 선행 과제.
- 자동 생성 내용의 사실성(일정·스펙 등)을 담당이 검증하지 않고 제출할 위험 → 핵심 수치는 "[확인 필요]" 표시로 출력.
